1. Deployment architecture — pick a capture model

TooVix DAM captures database activity three ways, all feeding one control plane. Which model applies depends on the engine and where it runs — a self-managed database (VM / on-prem) exposes all three; a managed PaaS instance you don't control exposes only Agentless.

Why eBPF is MySQL / PostgreSQL only. Host mode hooks SSL_read/SSL_write on OpenSSL (libssl). SQL Server (SChannel) and Oracle (libnnz) ship their own crypto stacks — there is no symbol to hook, so eBPF is not applicable, not merely unbuilt. For those engines, AgentLite is the primary path.
Everything converges. Whichever model you pick, events land in the same ingest and the same per-tenant event store — so classification, detection, and the tamper-evident audit trail work identically regardless of how the activity was captured.

2. AgentLite — enable the native telemetry (per engine)

AgentLite is a lightweight forwarder that reads telemetry the database already produces and ships each statement to DAM. It comes in two shapes:

  • Log tailer — tails a text log on the DB host (MySQL general log, PostgreSQL statement log).
  • Remote poller — polls the database's own audit telemetry over its protocol; nothing on the DB host, runs on any Linux box with network reach (SQL Server over TDS, Oracle over SQL*Net, MongoDB over the wire).
DB telemetry → AgentLite forwarder → TooVix DAM → Database Activity view (MySQL/PG log · SQL Server audit · Oracle (HTTPS, or an audit stream) UNIFIED_AUDIT_TRAIL · MongoDB profiler)

✅ transport-independent (reads after TLS decryption, sees local/IPC) · ✅ outbound-only (works on private, no-public-IP DBs) · ⚠️ detective only (cannot block — use the inline proxy) · ⚠️ records all SQL (treat logs as sensitive — see §12).

Requirements: Linux x86-64 for the collector · MySQL 5.7/8.x · MariaDB 10.x · PostgreSQL 12+ · SQL Server 2017+ · Oracle 19c/23ai · MongoDB 4.4+ · outbound HTTPS (443) to your DAM URL. Enable the source for your engine below — you'll pass the settings shown into §5.

🐬 MySQL / MariaDB — general query log

As MySQL admin, write the general log to a FILE at a fixed path:

-- runtime (takes effect now):
SET GLOBAL log_output      = 'FILE';
SET GLOBAL general_log_file = '/var/log/mysql/general.log';
SET GLOBAL general_log      = 'ON';

Persist it across restarts (the SET GLOBAL above is runtime-only):

# Debian/Ubuntu: /etc/mysql/mysql.conf.d/  ·  RHEL/Rocky: /etc/my.cnf.d/
sudo tee /etc/mysql/mysql.conf.d/zz-toovix-audit.cnf >/dev/null <<'EOF'
[mysqld]
log_output       = FILE
general_log      = 1
general_log_file = /var/log/mysql/general.log
EOF

Agent settings → DB_ENGINE=mysql · AUDIT_SOURCE=general_log · AUDIT_LOG=/var/log/mysql/general.log · TARGET_PORT=3306

🐘 PostgreSQL — statement logging

AgentLite parses PostgreSQL's standard statement log. Turn on statement logging and set the exact log_line_prefix the parser expects, then reload — no restart needed:

sudo -u postgres psql -c "ALTER SYSTEM SET log_statement = 'all';"
sudo -u postgres psql -c "ALTER SYSTEM SET log_line_prefix = '%m [%p] %u@%d ';"
sudo -u postgres psql -c "SELECT pg_reload_conf();"
The log_line_prefix matters. AgentLite matches lines shaped <time> [pid] user@db LOG:  statement: … — if your prefix differs, statements won't be parsed. This uses PostgreSQL's built-in logging; the pgaudit extension is not required.

The file is the cluster's main log, e.g. /var/log/postgresql/postgresql-16-main.log (match your PG major version). Agent settings → DB_ENGINE=postgresql · AUDIT_SOURCE=pgaudit · AUDIT_LOG=/var/log/postgresql/postgresql-<ver>-main.log · TARGET_PORT=5432

🪟 SQL Server — AgentLite runs remotely (nothing on Windows)

SQL Server's telemetry is binary, not a text log — so instead of tailing a file, AgentLite polls it over TDS. That's a bonus: the collector is a Linux container that only needs network reach to port 1433, so nothing is installed on the Windows DB host. Pick one of two sources:

SourceAUDIT_SOURCEScopingRow counts
SQL Server Auditsql_server_auditobject-level — audit only your tables (very clean)❌
Extended Eventsxeventsstatement-level (agent filters sys.* noise)✅ yes

Option 1 — SQL Server Audit (clean scoping, no row counts). Scope it to your tables with a database audit spec on the schema — a DATABASE-wide spec also captures sys.* reads:

CREATE SERVER AUDIT ToovixAudit TO FILE (FILEPATH='C:\SQLAudit\', MAXSIZE=50 MB, MAX_ROLLOVER_FILES=5) WITH (ON_FAILURE=CONTINUE);
ALTER SERVER AUDIT ToovixAudit WITH (STATE=ON);
-- in your database: audit the dbo schema only (excludes sys / INFORMATION_SCHEMA)
CREATE DATABASE AUDIT SPECIFICATION ToovixDbAudit FOR SERVER AUDIT ToovixAudit
  ADD (SELECT, INSERT, UPDATE, DELETE ON SCHEMA::dbo BY public) WITH (STATE=ON);

Agent → AUDIT_SOURCE=sql_server_audit · AUDIT_LOG=C:\SQLAudit\*.sqlaudit

Option 2 — Extended Events (adds row counts, which unlock the bulk-read / large-result policies):

CREATE EVENT SESSION ToovixXE ON SERVER
  ADD EVENT sqlserver.sql_statement_completed (
    ACTION (sqlserver.server_principal_name, sqlserver.client_hostname, sqlserver.database_name)
    WHERE ([sqlserver].[database_name]=N'YourDB'))
  ADD TARGET package0.event_file (SET filename=N'C:\SQLAudit\ToovixXE.xel', max_file_size=50, max_rollover_files=5)
  WITH (MAX_DISPATCH_LATENCY=5 SECONDS, STARTUP_STATE=ON);
ALTER EVENT SESSION ToovixXE ON SERVER STATE = START;   -- START, not ON

Agent → AUDIT_SOURCE=xevents · AUDIT_LOG=C:\SQLAudit\ToovixXE*.xel

Both sources need a DB login (the agent reads telemetry over TDS, not from disk): set DB_USER/DB_PASSWORD with CONTROL SERVER (Audit) or VIEW SERVER STATE (XEvents). Run the collector on any Linux host that can reach <db>:1433 — set TARGET_HOST to the SQL Server's address, which is how DAM identifies the instance.

Azure SQL / Managed Instance (PaaS) can't host any collector — use Agentless instead (database-level Auditing → Event Hub, consumed by the DAM connector; verified ~2-min end-to-end). Must be database-level — server-level auditing with a DB-scoped diagnostic setting captures nothing.

Classification works for SQL Server either way — it uses a read-only login over TDS, not the audit trail.

🍃 MongoDB — database profiler

MongoDB Community has no audit log at all (auditing is an Enterprise/Atlas feature), so there is no file to tail. The equivalent source is the built-in database profiler: with profiling on, mongod writes one document per operation into the capped collection <db>.system.profile, and AgentLite polls that collection over the wire. Like SQL Server, this means the collector needs a DB login but does not have to run on the DB host — so the same path covers Atlas and any remote mongod.

Enable profiling for the database you want monitored:

use <db>
// level 2 = every operation, slowms 0 = don't filter by duration
db.setProfilingLevel(2, { slowms: 0 })

Persist it across restarts (the command above is runtime-only, exactly like MySQL's SET GLOBAL):

# /etc/mongod.conf
operationProfiling:
  mode: all
  slowOpThresholdMs: 0

Create the monitoring login. Enabling the profiler needs the enableProfiler action — don't grant dbAdmin for it. dbAdmin bundles dropCollection and dropIndex, so a monitoring account holding it could destroy data. Create a custom role with just the one action instead:

// 1. a role that can ONLY toggle profiling
use <db>
db.createRole({
  role: 'toovixProfiler',
  privileges: [ { resource: { db: '<db>', collection: '' }, actions: [ 'enableProfiler' ] } ],
  roles: []
})

// 2. the monitoring user
use admin
db.createUser({
  user: 'dam_svc',
  pwd: '<strong-password>',
  roles: [
    { role: 'clusterMonitor',  db: 'admin' },  // read profiling status
    { role: 'read',            db: '<db>' },   // read system.profile + data
    { role: 'toovixProfiler',  db: '<db>' }    // set the profiling level, nothing else
  ]
})
Verified: this role set can read data, read system.profile and call setProfilingLevel — and is denied create, drop and write. If you'd rather not create a custom role, enable profiling server-side (operationProfiling.mode: all), set MONGO_AUTO_PROFILE=false, and drop the third role entirely — then read + clusterMonitor is all the agent needs.

Agent → DB_ENGINE=mongodb · AUDIT_SOURCE=profiler · TARGET_PORT=27017 · DB_NAME=<db> · DB_USER/DB_PASSWORD. No AUDIT_LOG — the source is the database connection, not a file.

Extra settingDefaultWhat it does
MONGO_URIbuilt from host/port/userFull connection string; use for Atlas (mongodb+srv://…) or custom TLS/replica-set options. Overrides the others.
MONGO_AUTH_SOURCEadminAuth database, if your user lives elsewhere.
AUDIT_POLL_SEC10How often system.profile is polled.
MONGO_AUTO_PROFILEtrueLet the agent enable profiling at startup. false to manage it server-side only.
MONGO_INCLUDE_GETMOREfalseSee the row-count warning below.
⚠️ Three limits specific to the profiler — read before relying on this trail.
  1. system.profile is capped (1 MB default, ≈ a few thousand operations). On a busy instance it wraps faster than the poll interval and operations are lost — profiler capture is best-effort, not a guaranteed audit trail. Enlarge it if that matters: db.setProfilingLevel(0), db.system.profile.drop(), db.createCollection('system.profile', { capped: true, size: 100000000 }), then re-enable.
  2. row_count under-reports large reads by default. A find reports only its first batch (101 documents), so a 10,000-document read looks like 101 rows. Set MONGO_INCLUDE_GETMORE=true to emit every cursor batch and get true read volume — at roughly one extra event per 100 documents. Turn this on if you use bulk-read/exfiltration policies.
  3. Profiling costs write throughput on the server, and the agent's own polling consumes space in the capped collection. A longer AUDIT_POLL_SEC reduces both.
Statements appear in Mongo syntax, not SQL — db.users.find({"email":"…"}) — because that's what actually ran. Operations still map onto the normal taxonomy (SELECT/INSERT/UPDATE/DELETE/DDL/GRANT), so existing policies and views work unchanged.

🔴 Oracle — Unified Auditing (polled over SQL*Net)

Oracle's telemetry is the Unified Audit Trail — the view UNIFIED_AUDIT_TRAIL in the AUDSYS schema. Like SQL Server, there is no text log; AgentLite polls the view over SQL*Net, so the collector runs on any Linux host that can reach port 1521 — nothing installs on the DB host, and the same collector covers Oracle on a VM, on RDS, and on OCI.

Turn on unified auditing (23ai is unified by default) and create a policy that captures DML. Scoping the policy BY <appuser> keeps Oracle's own recursive/background activity out of the trail:

CREATE AUDIT POLICY toovix_dml ACTIONS SELECT, INSERT, UPDATE, DELETE;
AUDIT POLICY toovix_dml BY appuser;   -- audit the application schema owner

Create the collector's read-only login. AUDIT_VIEWER reads UNIFIED_AUDIT_TRAIL; SELECT_CATALOG_ROLE lets it read V$SQLSTATS (for row counts) and DBA_TAB_COLUMNS (for classification). It gets no data access beyond that:

CREATE USER dam_svc IDENTIFIED BY "<strong-password>";
GRANT CREATE SESSION TO dam_svc;
GRANT AUDIT_VIEWER TO dam_svc;
GRANT SELECT_CATALOG_ROLE TO dam_svc;

Agent → DB_ENGINE=oracle · TARGET_PORT=1521 · DB_NAME=<service/PDB> (e.g. FREEPDB1) · DB_USER=dam_svc/DB_PASSWORD. No AUDIT_LOG — the source is the connection. Set TARGET_HOST to the database's address, which is how DAM identifies the instance.

Row counts come from a join to V$SQLSTATS on the statement text — exact for ad-hoc queries (EXECUTIONS=1), a per-execution average for parameterised app traffic. RDS Oracle works through this same path (the master user can read the audit views); OCI Autonomous also works but needs a wallet, so set a full oracle:// DSN with the wallet options. FGA (fine-grained auditing) is Enterprise-Edition only — Unified Auditing above covers Standard Edition and 23ai Free.
Verify (MySQL/PG): sudo tail -f <AUDIT_LOG> and run a query elsewhere — lines should appear. AgentLite tails the live file and handles rotation/truncation automatically; ensure logrotate + disk headroom on busy DBs.
Verify (MongoDB): run a query, then db.system.profile.find().sort({ts:-1}).limit(5) — documents should appear.
Verify (SQL Server / Oracle): run a query, then query the audit source directly (sys.fn_get_audit_file / UNIFIED_AUDIT_TRAIL) — rows should appear within the poll interval (~10s).

3. Classification reader (optional)

DAM can classify which columns hold PII/PCI. This is separate from capture — it logs into the database as a least-privilege reader. Skip this if you only want activity capture.

MySQL / MariaDB:

CREATE USER 'dam_svc'@'%' IDENTIFIED BY '<strong-password>';
GRANT SELECT, PROCESS ON *.* TO 'dam_svc'@'%';
FLUSH PRIVILEGES;

PostgreSQL:

CREATE USER dam_svc WITH PASSWORD '<strong-password>';
GRANT pg_monitor TO dam_svc;
GRANT CONNECT ON DATABASE <db> TO dam_svc;
\c <db>
GRANT USAGE ON SCHEMA public TO dam_svc;
GRANT SELECT ON ALL TABLES IN SCHEMA public TO dam_svc;

SQL Server: a login with VIEW ANY DEFINITION (reads information_schema across databases) — or reuse the capture login if it already has CONTROL SERVER/VIEW SERVER STATE.

Oracle: the dam_svc user from §2 already holds SELECT_CATALOG_ROLE, which reads DBA_TAB_COLUMNS — no extra grant needed. Set DB_NAME=<service/PDB>.

MongoDB: the read role from §2 is sufficient — classification samples documents per collection to infer the field set (there is no fixed schema to read).

dam_svc is read-only — it inspects schema/metadata but can never modify data. Pass it as DB_USER/DB_PASSWORD with CLASSIFY=true (PostgreSQL / SQL Server / Oracle / MongoDB also need DB_NAME).

How classification reads each engine: MySQL/PostgreSQL/SQL Server → information_schema.columns; Oracle → DBA_TAB_COLUMNS (scoped to non-Oracle-maintained schemas); MongoDB → document sampling (100 docs/collection, nested fields to depth 3). Mongo sampling is probabilistic — a sensitive field present in only a few rare documents can be missed, so its confidence is reported lower than the SQL engines'.

4. Enrollment token

In the TooVix DAM console:

  • Go to Agents → Deploy monitoring (or Databases → register your instance, then Deploy).
  • Select your instance and the AgentLite (Audit Forwarder) mode.
  • Copy the enrollment token (looks like tvxenr_…) and the control-plane URL (e.g. https://dam.yourcompany.com).

The token binds the agent to your workspace/tenant, so captured activity is attributed correctly. The console also generates a ready-to-paste command — this guide explains each part so you can do it by hand.

5. Install & run the agent

Pick one install method. The agent needs to read the audit source (§2) and reach DAM over HTTPS. The examples below use MySQL values; for the other engines swap the settings shown per engine in §2 (a summary table follows the examples). Set these once:

CONTROL_PLANE="https://dam.yourcompany.com"          # from §4
ENROLL_TOKEN="tvxenr_xxxxxxxxxxxxxxxxxxxx"            # from §4
AUDIT_LOG="/var/log/mysql/general.log"               # from §2
DB_VM_HOST="10.0.0.10"                               # THIS VM's private IP / hostname
DB_VM_HOST is how DAM identifies this database instance — use the VM's private IP or hostname, matching the instance as registered in the console. It must be unique per DB VM (don't use 127.0.0.1, or several VMs collapse into one record).

Option A — Docker (simplest)

docker run -d --name toovix-agent-audit --restart unless-stopped \
  --user 0 \
  -v ${AUDIT_LOG}:${AUDIT_LOG}:ro \
  -e MODE=audit-forward \
  -e DB_ENGINE=mysql \
  -e TARGET_HOST=${DB_VM_HOST} \
  -e TARGET_PORT=3306 \
  -e AUDIT_SOURCE=general_log \
  -e AUDIT_LOG=${AUDIT_LOG} \
  -e AGENT_ENROLL_TOKEN=${ENROLL_TOKEN} \
  -e CONTROL_PLANE=${CONTROL_PLANE} \
  <your-dam-agent-image>            # e.g. registry.yourcompany.com/dam-agent:latest

Option B — Native package (.deb / .rpm) + systemd

# Download the agent from your control plane
curl -fsSL ${CONTROL_PLANE}/api/download/dam-agent_amd64.deb -o dam-agent.deb   # Debian/Ubuntu
sudo dpkg -i dam-agent.deb
# RHEL/Rocky:  curl -fsSL ${CONTROL_PLANE}/api/download/dam-agent_amd64.rpm -o dam-agent.rpm && sudo dnf install -y ./dam-agent.rpm

# Configure this instance (templated unit: dam-agent@<name> reads /etc/toovix/agent-<name>.env)
sudo mkdir -p /etc/toovix
sudo tee /etc/toovix/agent-audit.env >/dev/null <<EOF
MODE=audit-forward
DB_ENGINE=mysql
TARGET_HOST=${DB_VM_HOST}
TARGET_PORT=3306
AUDIT_SOURCE=general_log
AUDIT_LOG=${AUDIT_LOG}
AGENT_ENROLL_TOKEN=${ENROLL_TOKEN}
CONTROL_PLANE=${CONTROL_PLANE}
EOF

sudo systemctl enable --now dam-agent@audit
journalctl -u dam-agent@audit -f     # watch it enroll + start tailing
To also run classification (§3), add CLASSIFY=true, DB_USER=dam_svc, DB_PASSWORD=<the password> to the env / -e flags.

MongoDB variant

There's no log file, so drop AUDIT_LOG and the -v log mount, and supply the connection instead. The collector needs no privileged access to the host and can run anywhere that can reach 27017:

docker run -d --name toovix-agent-audit --restart unless-stopped \
  -e MODE=audit-forward \
  -e DB_ENGINE=mongodb \
  -e AUDIT_SOURCE=profiler \
  -e TARGET_HOST=${DB_VM_HOST} \
  -e TARGET_PORT=27017 \
  -e DB_NAME=<db> \

  -e AGENT_ENROLL_TOKEN=${ENROLL_TOKEN} \
  -e CONTROL_PLANE=${CONTROL_PLANE} \

# Atlas / replica sets: replace TARGET_* and DB_USER/DB_PASSWORD with
#   -e MONGO_URI='mongodb+srv://user:pass@cluster.xxxxx.mongodb.net/'
# but keep TARGET_HOST set — it's how DAM identifies the instance.

SQL Server & Oracle variant (remote poller — nothing on the DB host)

6. Verify

  • Run a distinctive query against MySQL over TCP, so it's logged: 
    mysql -h 127.0.0.1 -u <someuser> -p -e "SELECT 'dam-verify-123', NOW();"
  • In the console, open Databases → your instance → Database Activity (in the same workspace the enrollment token belongs to).
  • Within a few seconds you should see SELECT 'dam-verify-123'… attributed to your DB and principal.

If it appears — you're done. AgentLite is capturing.

7. Agent — network · host eBPF · inline proxy

The Agent taps the connection instead of reading the database's audit. It's for MySQL and PostgreSQL — open protocols on OpenSSL hosts — and is how you get exact row counts on every event and, with the proxy, real-time blocking. All three run from the same agent image, selected by MODE. Deploy on the DB host (network/host) or in the connection path (proxy).

ModeMODESeesPath change?Blocks?
NetworknetworkCleartext wire traffic (passive, AF_PACKET). Blind to TLS.No — out of bandNo
Host (eBPF)hostBelow TLS — hooks SSL_read/SSL_write, so it reads encrypted sessions. Blind to cleartext (pair with network).No — transparentNo
Inline proxyproxyTerminates the connection, sees the real end-user, forwards to the DB.Yes — clients connect to the proxyYes
Only MySQL / PostgreSQL. Host eBPF hooks OpenSSL — SQL Server (SChannel) and Oracle (libnnz) can't be hooked, and MongoDB wire-decode is low-value. For those engines use AgentLite (§2). Network mode can passively decode SQL Server's TDS, but SQL Server encrypts by default, so it captures little in practice.

Network agent (passive wire capture)

Runs on the DB host, shares its network namespace, and needs NET_ADMIN/NET_RAW to open a raw socket. Zero path change, ~0 overhead:

docker run -d --name toovix-agent-network --restart unless-stopped \
  --network host --cap-add NET_ADMIN --cap-add NET_RAW \
  -e MODE=network \
  -e DB_ENGINE=mysql -e TARGET_PORT=3306 \
  -e TARGET_HOST=${DB_VM_HOST} \
  -e AGENT_ENROLL_TOKEN=${ENROLL_TOKEN} -e CONTROL_PLANE=${CONTROL_PLANE} \
  <your-dam-agent-image>
# PostgreSQL: DB_ENGINE=postgresql TARGET_PORT=5432

Host agent (eBPF, below TLS)

Also on the DB host, but attaches eBPF uprobes to the database's libssl — so it reads TLS-encrypted sessions the network agent can't. Needs CAP_BPF/CAP_PERFMON (kernel 5.8+) and the host PID namespace:

docker run -d --name toovix-agent-host --restart unless-stopped \
  --network host --pid host --privileged \
  -e MODE=host \
  -e DB_ENGINE=postgresql -e TARGET_PORT=5432 \
  -e TARGET_HOST=${DB_VM_HOST} \
  -e AGENT_ENROLL_TOKEN=${ENROLL_TOKEN} -e CONTROL_PLANE=${CONTROL_PLANE} \
  <your-dam-agent-image>
Deploy network + host together for full coverage — they're complementary, not redundant: network reads cleartext (TLS is opaque to it), host reads encrypted (cleartext never calls SSL_read). Neither double-counts the other. This is the recommended "full visibility" posture for self-managed MySQL/PG.

Inline proxy (the only mode that can block)

The proxy listens on its own port, forwards to the real database, and can drop or kill a session per policy. Clients must connect to the proxy instead of the DB — the one mode with a connection-path change. Run it wherever it can reach the DB, then repoint your apps:

docker run -d --name toovix-agent-proxy --restart unless-stopped \
  -p 3307:3307 \
  -e MODE=proxy \
  -e DB_ENGINE=mysql \
  -e LISTEN_PORT=3307 \
  -e UPSTREAM=<db-host>:3306 \
  -e TARGET_HOST=${DB_VM_HOST} \
  -e AGENT_ENROLL_TOKEN=${ENROLL_TOKEN} -e CONTROL_PLANE=${CONTROL_PLANE} \
  <your-dam-agent-image>
# apps now connect to the proxy host on :3307 instead of the DB on :3306
Pair the proxy with a network agent so clients that bypass the proxy and hit the DB directly are still seen. Note: if the proxy→DB hop is cleartext, the network agent records that leg too, and its client_ip is the proxy, not the end user — keep the proxy→DB hop on TLS to avoid that duplicate.

8. Agentless — managed / PaaS (cloud audit stream)

For a managed database you don't control — RDS, Cloud SQL, Azure SQL, Cosmos — you can't install any agent. Instead the cloud emits the database's native audit to a managed stream, and the DAM connector consumes it. Nothing runs on the host; setup is entirely on the cloud side plus one connector in DAM.

Managed DB → native audit → cloud stream → DAM connector → Database Activity view (RDS · Cloud SQL · (Kinesis · Pub/Sub · Azure SQL · Cosmos) Event Hub)
Agentless ≠ AgentLite publishing to a stream. In Agentless the cloud produces the audit records natively (you never touch the host). Below, each cloud's setup — pick the row matching your database.

🟦 Azure SQL Database → Event Hub (verified ~2-min end-to-end)

Enable database-level auditing to an Event Hub, then point the DAM connector at it. It must be database-level — server-level auditing with a DB-scoped diagnostic setting captures nothing.

az sql db audit-policy update -g <RG> -s <server> -n <db> \
  --state Enabled --event-hub-target-state Enabled \
  --event-hub <hub> --event-hub-authorization-rule-id <send-rule-id> \
  --actions BATCH_COMPLETED_GROUP SUCCESSFUL_DATABASE_AUTHENTICATION_GROUP

In DAM, add an Azure Event Hub connector with the hub's listen connection string and your tenant's enrollment token. Captures statements, logins, and audit-config changes.

🟨 GCP Cloud SQL (MySQL / PostgreSQL) → Pub/Sub (verified)

Enable the database's audit flags, route its logs to a Pub/Sub topic via a Cloud Logging sink, and let the DAM connector pull the subscription.

# 1. turn on audit (PostgreSQL example — pgaudit)
gcloud sql instances patch <instance> \
  --database-flags=cloudsql.enable_pgaudit=on,pgaudit.log=read,write,ddl,role
# MySQL: --database-flags=cloudsql_mysql_audit=on  (then create an audit rule, wildcard '*')

# 2. route audit logs → Pub/Sub topic
gcloud pubsub topics create toovix-dam-audit
gcloud logging sinks create toovix-cloudsql-audit-sink \
  pubsub.googleapis.com/projects/<proj>/topics/toovix-dam-audit \
  --log-filter='resource.type="cloudsql_database" AND resource.labels.database_id="<proj>:<instance>"'
# grant the sink's writer identity roles/pubsub.publisher on the topic, then create a subscription

In DAM, add a GCP Pub/Sub connector pointing at the subscription. Cloud SQL emits a structured audit proto (not text), which the connector normalizes.

🟧 AWS RDS / Aurora → Kinesis (roadmap)

RDS publishes a Database Activity Stream (or exports the audit/general log to CloudWatch Logs) which a subscription filter forwards to Kinesis. The Kinesis source adapter is on the roadmap; for MySQL/PostgreSQL on RDS today, the AgentLite remote poller or an EC2-hosted collector is the interim path. RDS Oracle is reachable by the AgentLite Oracle collector (§2) via the master user.

🍃 MongoDB — Cosmos / Atlas (roadmap)

Azure Cosmos DB for MongoDB emits MongoRequests diagnostics to an Event Hub (the transport already exists — a normalizer is pending). MongoDB Atlas has no native push stream; its audit is a pull-based log export to S3. For managed MongoDB today, the AgentLite profiler poller (§2) can target a remote mongod including Atlas.

The core steps are identical on GCP, AWS, and Azure. The only cloud-specific concern is giving the VM outbound HTTPS (443) to reach DAM — especially for private (no public IP) databases.

10. (Optional) AgentLite → audit stream instead of direct HTTPS

This is not Agentless (§8). Here the AgentLite agent — which you installed — publishes to a stream instead of POSTing DAM over HTTPS, as a durable buffer that survives brief DAM outages. Agentless (§8) is when the cloud itself emits the audit and no agent exists. Direct HTTPS (§5) is the simplest default; use this only if you want the buffer.

The bus is the cloud-native stream for your provider:

⚠️ The stream is never automatic. On every cloud you provision the stream yourself and grant access — a fresh database or VM emits nothing to a stream until you configure it. There are always two parts: (1) create the stream (a Pub/Sub topic, Kinesis stream, or Event Hub), and (2) grant the publisher (this agent's VM identity) and the DAM consumer access to it.
CloudAudit streamPublish permission on the VM's identity
GCPCloud Pub/Sub topicroles/pubsub.publisher on the topic
AWSKinesis stream (or SNS/SQS)an IAM policy allowing PutRecord on the stream
AzureEvent Hubthe Azure Event Hubs Data Sender role

Coordinate with your DAM operator — they provide the stream/topic name and run the subscription consumer (dam-audit-consumer). The agent authenticates with the VM's own attached cloud identity (metadata server / managed identity) — no key file, since SA/secret keys are often disabled by org policy. Then add to the agent env:

AUDIT_TOPIC=<stream-or-topic-name>     # e.g. toovix-dam-audit
GCP_PROJECT=<project-id>               # optional — auto-detected from the metadata server

When AUDIT_TOPIC is set the agent publishes to the stream; when unset it POSTs the control plane over HTTPS. Everything else stays the same.

Provision the stream and grant the VM's own identity publish access — examples for each cloud. The agent authenticates through the VM's metadata service (attached SA / instance role / managed identity), so no key files:

GCP — Cloud Pub/Sub

gcloud pubsub topics create toovix-dam-audit
gcloud pubsub topics add-iam-policy-binding toovix-dam-audit \
  --member="serviceAccount:<VM_SERVICE_ACCOUNT_EMAIL>" --role="roles/pubsub.publisher"
# attach the SA to the VM (requires stop/start):
gcloud compute instances stop <VM> --zone <ZONE>
gcloud compute instances set-service-account <VM> --zone <ZONE> \
  --service-account=<VM_SERVICE_ACCOUNT_EMAIL> --scopes=cloud-platform
gcloud compute instances start <VM> --zone <ZONE>
# agent env →  AUDIT_TOPIC=toovix-dam-audit   GCP_PROJECT=<project-id>  # (project optional)

AWS — Kinesis Data Stream

aws kinesis create-stream --stream-name toovix-dam-audit --shard-count 1
# let the EC2 instance role publish (agent uses the instance profile via IMDS — no keys):
aws iam put-role-policy --role-name <EC2_INSTANCE_ROLE> --policy-name toovix-dam-publish \
  --policy-document '{"Version":"2012-10-17","Statement":[{"Effect":"Allow",
    "Action":["kinesis:PutRecord","kinesis:PutRecords"],
    "Resource":"arn:aws:kinesis:<REGION>:<ACCOUNT_ID>:stream/toovix-dam-audit"}]}'
# agent env →  AUDIT_TOPIC=toovix-dam-audit   AWS_REGION=<region>

Azure — Event Hub

az eventhubs namespace create -g <RG> -n <NAMESPACE> --sku Standard
az eventhubs eventhub create  -g <RG> --namespace-name <NAMESPACE> -n toovix-dam-audit
# give the VM's managed identity the Data Sender role (agent uses managed identity via IMDS — no keys):
az role assignment create --assignee <VM_MANAGED_IDENTITY_OBJECT_ID> \
  --role "Azure Event Hubs Data Sender" \
  --scope "/subscriptions/<SUB>/resourceGroups/<RG>/providers/Microsoft.EventHub/namespaces/<NAMESPACE>/eventhubs/toovix-dam-audit"
# agent env →  AUDIT_TOPIC=toovix-dam-audit   AZURE_EVENTHUB_NAMESPACE=<namespace>
Implementation status: the agent's built-in AUDIT_TOPIC publisher ships for GCP Pub/Sub today; the AWS (Kinesis) and Azure (Event Hub) blocks show the equivalent provisioning and are the roadmap for those streams. On all three clouds right now, the direct-HTTPS path (§5) works with no stream at all — use it unless your DAM operator runs a stream consumer for your cloud.
Two ways the audit reaches the stream. With AgentLite on a VM (this guide), the on-host forwarder publishes directly to the stream — no cloud log sink is involved. For a managed / PaaS database (the Agentless path) there's no host to install on, so the cloud itself routes the database's native audit into the stream — e.g. a Cloud SQL log sink → Pub/Sub, an RDS Database Activity Stream → Kinesis, or an Azure SQL diagnostic setting → Event Hub. Either way, DAM just consumes the stream.

12. Security & privacy notes

  • The general query log contains every SQL statement (from all apps and users) in cleartext on the VM. Restrict file permissions, enable rotation, and treat it as sensitive data.
  • AgentLite is detective, not preventive — it alerts after the fact and cannot block. For real-time blocking, use Inline Proxy mode.
  • The agent only makes outbound connections to DAM (HTTPS, or your audit stream). No inbound ports are opened; DAM never connects into your database network.
  • Least privilege: the optional dam_svc user is read-only (SELECT, PROCESS). Rotate its password per your policy.
  • MongoDB: system.profile contains query filters and inserted documents, so it holds real data values on the server — treat the profiler as sensitive, and note that enabling it is a visible change to the database's behaviour (write overhead), not a passive read. Keep dam_svc on the toovixProfiler custom role from §2: dbAdmin would let a monitoring account drop collections, which is far more privilege than capture requires.
Questions, or an engine not covered here (Oracle / others)? Contact your TooVix DAM operator. AgentLite audit-forward supports MySQL/MariaDB, PostgreSQL, SQL Server, and MongoDB; other engines use network, host, or inline-proxy capture.
